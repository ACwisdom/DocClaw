普朗医疗「DoctorClaw」项目现状与目标详细梳理
一、项目定位与核心概念
1.1 项目名称与定位
项目名称：DoctorClaw（类比 OpenClaw 的命名逻辑）
公司主体：普朗医疗
战略定位：基于国产算力的场算力应用，专门针对医生群体展开需求设计
产品形态：医生的”数字分身”（DoctorClaw），不局限于单一网页形态，支持多种交互形式
1.2 “医生分身”的双层概念
录音中明确提到医生分身包含两个概念维度：
第一维度：医生分身主（日常工作的数字化替身） - 覆盖医生完整工作流：诊前、诊中、诊后 - 以患者管理为核心（非简单的辅助工具） - 解决痛点：一个医生管理多个病患时的日常随访、管理工作负担 - 功能形态：通过分身自动/半自动管理患者，替代医生完成重复性事务
第二维度：任务执行能力 - 实时任务：即时响应的患者咨询、异常处理等 - 计划性任务：随访计划、定期触达、长期咨询指导等定时/周期性工作
二、核心差异化与功能架构
2.1 核心差异化定位（与现有医院系统的本质区别）
何总反复强调：患者管理是核心，其他基础能力只是”标配”
现有医院系统已经具备的能力（普朗医疗认为是”基础标配”）： - 常规的医院信息化功能（HIS系统基础能力） - 已有的诊中辅助功能
DoctorClaw 的核心聚焦点： 1. 患者管理（装科、装病、装人的精细化管理体系） 2. 触达用户：主动的患者触达机制（非被动等待患者上门） 3. 异常监控：对患者状态的异常识别与预警 4. 长期咨询指导：持续性的医患互动，非一次性问诊 5. 随访计划：结构化的随访管理体系
2.2 产品形态与交互要求
不局限于网页：明确提到”产品表现形式不局限于说一个一个网页的”
交互形式：需要支持语音交互（提到 Google Speech）、实时对话
部署场景：医生工作场景的一体机/有线算力服务器（非纯云端API调用）
客户端需求：需要采集医院医生的实际使用需求，快速通过 Lark coding 产出产品原型
三、技术架构与基础设施
3.1 算力基础（国产算力方向）
算力类型：场算力（有线算力上的一体机或服务器，非云端API）
技术挑战：团队需要了解：
什么样的算力规模可以支撑多大参数的模型
模型的吞吐率表现
国产算力生态的适配（区别于常用的云端AI API）
3.2 核心技术栈要求
面试中明确提出的技术能力要求，反映项目技术路线：
基础必备技能： - RAG 技术：知识库检索增强（用于医生知识库、患者资料管理） - 训练微调：模型微调能力 - Skill 框架：技能编排 - MCP（Model Context Protocol）：模型上下文协议，用于工具调用和外部系统集成
产品快速交付工具： - Lark coding / Lark code：快速产品原型开发 - Pencil：产品交互设计 - Google Speech：语音交互能力 - Web code：产品交付
开发模式： - 从需求采集 → Lark coding 快速出产品 → 产品交互设计 → Web code 交付的闭环 - 强调”快速形成产品交互”和”快速交付”
